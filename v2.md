# v2 Specifications

## Context

Requirements gathered from client meeting after demo of the PoC.

- Standardize the names of the specs and doubts files.
- Add a detailed documentation based on the specs including a definition of the object model (uml class diagram).


---

## 1. Terminology

- Replace all uses of "goalkeeper" / "keeper" with **"goalie"** throughout the application (UI labels, error messages, database column names if applicable).
- Player roles should be referred to as **"skater"** and **"goalie"** (not "field player" or "keeper").

---

## 2. Player Jersey Numbers

- Each player has a **jersey number** that is tracked **per season** (not a fixed attribute on the player record).
- A jersey number is **not globally unique**: different players can wear the same number in different seasons.
- Within the same season, a **temporary/substitute player** may reuse a jersey number that is also assigned to another temporary player.
- Numbers can change between seasons (or, if needed, within a season for a given player).
- Data model implication: add a `player_season_numbers` table (or equivalent) with `player_id`, `season_id`, and `number`, rather than a single `number` column on the `players` table. <!-- TODO:  Wouldn't be a better approach to keep just the jersey number of each player in each match? -->

---

## 3. Roster and Player Editing

- Allow **editing** an existing player's details: name, primary role, and jersey number (for the current season).
- Editing must preserve all historical game stats linked to the player.

---

## 4. Simplified Player Roles

- A player has **one primary role** (`skater` or `goalie`) as a profile attribute.
- As an exception, a player may be recorded as playing a **different role in a specific game** (e.g., a skater filling in as goalie for one game).
- The exception-role entry should still produce the correct stat family for that game (skater stats or goalie stats), as per the current aggregation logic.
- UX goal: default the stat entry form to the player's primary role; allow manual override per game entry.

---

## 5. Shots Received as a Calculated Field

- `ShotsReceived` for a goalie must always equal `Saves + GoalsAgainst`; it is **not a free-entry field**.
- Remove the manual "Shots" input from the game-entry form for goalies.
- Display the computed `ShotsReceived` value in the interface (read-only) for transparency.
- This is consistent with the PoC fallback behaviour but now it must be the only supported path (no override allowed).

---

## 6. Read-Only Dashboard

- The season stats dashboard **must be read-only**: users should not be able to type or modify data directly in the dashboard view.
- Any corrections must go through the dedicated correction flow (see §9).

---

## 7. Improved Duplicate-Game Error Message

- When attempting to save a game that has already been recorded (same season + date + opponent), the error message must clearly state:
    - That a duplicate was detected.
    - Which game record already exists (date, opponent, season).
    - How to correct existing stats (direct the user to the correction flow).

---

## 8. Automatic Email on Game Save

- As soon as a game is successfully saved, the system must **automatically send** the season stats summary email to all mailing-list recipients.
- The current manual "Send" button may remain as a secondary option (resend), but it is no longer the only trigger.
- Email delivery result (success / failure) continues to be logged.

---

## 9. Correct Game Stats

- Provide a **"Correct last game"** shortcut button that loads the most recently saved game into an editable form.
- Also support correcting **any past game** (not only the last one): allow the user to select a game from a list (filtered by season) to load it for editing.
- On save, the corrected stats replace the previous ones atomically (within a transaction).
- After a correction, the automatic email (§8) should be triggered.

---

## 10. Regular Season vs. Playoff Stats

- Games must be tagged with a **game type**: `regular` or `playoff`.
- Season stats are **displayed separately** for regular season and playoffs (two distinct stat blocks in the dashboard).
- The data model requires a `game_type` field on the `games` table (default `regular` for backward compatibility).
- Aggregation and email summary must reflect the split.

---

## 11. UX Improvements (General)

- Overall UX should be kept as simple as possible for non-technical users.
- Reduce the number of required clicks / form fields for the most common workflow (entering a game result).
- Provide clear feedback messages for all user actions (success and error).

---

## Open Questions

- Assumption: one player can only play for a team in one season at a time, but in a different one in another season.
- **Jersey number uniqueness within a season**: can two *regular* (non-temporary) players share the same number in the same season? (Current assumption: no — only temporary players may share.)
- **Game correction permissions**: should any correction be allowed at any time, or only within a time window (e.g., 24 hours after the game)?
- **Playoff seasons**: are playoffs part of the same season label (e.g., `2024-25`) or a separate label? Decision needed before data-model changes.
- **Email trigger on correction**: should the email also fire when a game correction is saved, or only on first save?
- **Backward compatibility**: how should existing records without `game_type` or `player_season_numbers` be handled on first upgrade?
